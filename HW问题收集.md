## HW 问题收集

列举在HW 1、HW1.1过程里，你所遇到的2\~3个通过自己学习已经解决的问题，和2\~3个尚未解决的问题与挑战

### ### 已解决

1. **什么是“标量 Derived Store”及其作用？**
   * **上下文** ：Coding Agent 建议 UI 读取领域对象接口，仅保留少量标量 derived store。
   * **解决手段** ：通过查询资料了解到，这是为了避免在 Store 中存储庞大的对象或数组。通过 `derived` 导出布尔值（如 `canUndo`）或数字，可以利用 Svelte 的脏检查机制，只有当这些基础值变化时才触发组件重绘，从而提高渲染性能。
2. **领域模型（Sudoku.js）的防御性拷贝**
   * **上下文** ：Review 指出 `Sudoku` 在处理二维数组时有良好的自觉。
   * **解决手段** ：在 `getGrid()`、`clone()` 和 `toJSON()` 中均实现了二维数组的深拷贝，确保外部代码无法通过引用直接篡改领域模型内部的 `grid` 状态，维护了对象的封装性。
3. **基于快照的 Undo/Redo 逻辑实现**
   * **上下文** ：需要实现撤销/重做功能。
   * **解决手段** ：在 `Game` 聚合根中维护 `undoStack` 和 `redoStack`。每次执行 `guess()` 操作前，将当前局面快照压入撤销栈，并在执行 `undo/redo` 时通过快照恢复，确保了操作历史的清晰管理。

### ### 未解决与挑战

1. **UI 渲染与领域对象的“单一真相源”脱节**
   * **当前瓶颈** ：Review 发现 `Board/index.svelte` 仍在从旧的 `@sudoku/stores/grid` 获取状态，而非消费由 `Game` 对象同步出来的视图状态。
   * **挑战** ：如何彻底切断 UI 对旧 Store 的依赖，确保整个棋盘（渲染、输入、判定）完全基于领域对象导出的 `gameStore` 运行。
2. **数独业务逻辑入模与胜利判定漏洞**
   * **当前瓶颈** ：目前的 `Sudoku.guess()` 仅校验坐标，不校验行列宫冲突；`gameWon` 判定仅检查格子是否填满，不检查合法性。
   * **挑战** ：需要将数独的核心规则（冲突检测、不可修改原始数字）下沉到领域层，确保业务语义的正确性。
3. **聚合根（Game）的封装边界过弱**
   * **当前瓶颈** ：`Game` 直接持有并暴露了可变的 `Sudoku` 实例。外部代码可以通过 `getSudoku()` 直接调用其方法，从而绕过 `Game` 的操作历史管理。
   * **挑战** ：如何重新设计接口，隐藏内部实例，强制所有状态变更必须经过聚合根（`Game`）以维护一致性。
4. **Svelte 接入层的状态重置与序列化失真**
   * **当前瓶颈** ：开始新游戏时，UI 层的 `cursor`、`timer` 等状态未同步重置；且从 JSON 恢复游戏时，历史记录（undoStack）会丢失。
   * **挑战** ：需要完善 `Store Adapter` 的用例编排，确保在游戏生命周期切换和持久化恢复时，所有状态（包括历史数据）都能完整对齐。
5. **`sameArea` 变量的逻辑与来源**
   * **当前瓶颈** ：在 `Board/index.svelte` 中，`sameArea` 的计算逻辑（`isSameArea($cursor, x, y)`）在当前 CA 建议的重构下显得模糊。
   * **挑战** ：需要明确该逻辑是属于 UI 交互层（View Model）还是应该由领域对象提供判定接口。
